Three questions. One explicit path decision stated before Phase 2 starts.

Team lead / data / engineering / admin → deep dive. Specific immediate question / casual mention → lightweight.

Quirk philosophy

Quirks are the system's problem, not the user's.

Find them before the user does. Solve them. The 8-level hierarchy defines what "exhausted every option" actually means.

📄Full protocol → QUIRKS-PROTOCOL

The workaround hierarchy

Exhaust these in order. Step 8 requires a written record of why every earlier step failed.

1

Handle in sync.py client code

Intercept raw behavior, return clean data. User never knows the quirk existed.

Best
2

Reconstruct from other fields

Calculate from what is available. thumbstop_ratio = 3s_views / impressions when the ratio field is null.

Best
3

Use a different approach

Different endpoint, different query, different data source that avoids the limitation entirely.

Good
4

Chunk, batch, or paginate

Work around size and rate limits in the sync script. Transparent to user.

Good
5

Cache or pre-compute

SQLite with TTL. Minimize API calls. Avoid hitting the limitation repeatedly.

Good
6

Warn proactively and specifically

Fire a specific warning before the user hits the problem. Name the exact condition and exact workaround.

Acceptable
7

Ask a data-savvy user

Only if they have account-level info you genuinely need. Don't ask anyone else.

Rare
8

State it can't be done

Only after 1–7 exhausted, with a written record. Should be genuinely rare. If you get here, something went wrong earlier.

Last resort
handled-in-code

Zero user-visible impact. The best outcome.

handled-by-warning

Specific warning fires before user hits the problem. Acceptable when user-side action is required.

unhandled ⚠

A failure state. Pending trust failure. Open debt. Every workaround attempt must be documented.
